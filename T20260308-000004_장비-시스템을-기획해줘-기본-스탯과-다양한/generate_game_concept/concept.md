# generate_game_concept

# 게임 컨셉 초안: **리릭 포지 아레나** *(가제)*

## 한 줄 정의
**리릭 포지 아레나**는 소수의 장비에 강한 개성을 가진 옵션을 새기고, 그 옵션을 강화하거나 교체해 2분 전투의 성능을 검증하는 **아웃게임 빌드 중심 액션 RPG**다.

## 장르
- **아웃게임 빌드 중심 서바이버 액션 RPG**
- **장비 파밍 + 옵션 제작/재련 + 짧은 전투 검증형 구조**

## 핵심 테마
- 전투는 성장의 장소가 아니라 **준비한 빌드의 완성도를 시험하는 테스트 아레나**다.
- 플레이어는 단순한 전사가 아니라, 유물 장비를 해체하고 재구성하는 **전투 대장장이**다.
- 장비 수는 많지 않지만, 각 장비에 붙은 옵션 한 줄 한 줄이 전투 스타일을 결정한다.
- 따라서 이 게임의 재미는 “무엇을 많이 주웠는가”보다 **무엇을 어떻게 조합했는가**에 있다.

## 플레이어 판타지
- **적은 수의 장비를 완벽하게 조율해 강력한 빌드를 만든다.**
- **짧은 전투에서 내가 설계한 옵션 조합이 폭발적으로 작동하는 쾌감을 얻는다.**
- **원하는 옵션을 만들고, 더 강하게 키우고, 필요 없으면 갈아엎으며 이상적인 세팅을 추적한다.**

## 기본 게임 플레이 루프
1. 아웃게임 공방에서 장비 몇 개를 선택하고 기본 스탯과 옵션 방향을 설계한다.
2. 2분 내외의 전투에 진입해 생존, 처치, 점수로 빌드 성능을 검증한다.
3. 전투 결과에 따라 옵션 제작, 강화, 변경에 필요한 재료 아이템을 획득한다.
4. 공방으로 돌아와 옵션을 만들고, 강화하고, 교체하며 장비를 더 정교하게 다듬는다.
5. 다음 전투에서 더 빠른 처치, 더 안정적인 생존, 더 높은 점수를 노리며 파밍 목표를 갱신한다.

## 장비 시스템 컨셉
- 장비 슬롯은 **소수 정예**로 유지한다. 중요한 것은 착용 수가 아니라 **옵션 조합의 밀도**다.
- 각 장비는 **기본 스탯 정체성**을 가진다.
- 무기는 공격 성향, 방어구는 생존 성향, 코어는 빌드 방향성, 장신구는 유틸리티 성향을 담당한다.
- 장비 차별화의 핵심은 등급보다 **옵션 라인과 옵션 시너지**다.
- 옵션은 단순 수치 상승이 아니라 **태그 기반 플레이 성향**을 만든다.
- 예시 태그는 `투사체`, `치명타`, `출혈`, `연쇄`, `돌진`, `보호막`, `소환`, `범위` 등이다.
- 좋은 파밍의 목표는 더 많은 장비 확보가 아니라 **원하는 옵션 조합과 높은 옵션 품질**을 달성하는 것이다.

## 핵심 옵션 액션
- **옵션 만들기**: 빈 옵션 라인이나 해금된 라인에 원하는 계열 옵션을 각인한다.
- **옵션 강화하기**: 기존 옵션의 수치나 티어를 높여 핵심 빌드 축을 강화한다.
- **옵션 바꾸기**: 특정 옵션군 안에서 재추첨하거나 전체 라인을 변환해 빌드 방향을 수정한다.
- **옵션 고정하기**: 만족스러운 옵션은 잠가 두고 나머지 라인만 다시 조정한다.
- 이 구조 덕분에 장비 수가 적어도 플레이어는 계속해서 **“한 줄만 더 좋아지면 완성된다”**는 강한 파밍 목표를 가진다.

## 재료 아이템 방향
- **각인 파편**: 새 옵션을 만들 때 사용하는 기본 재료.
- **증폭 촉매**: 옵션 강화 전용 재료.
- **변환 프리즘**: 옵션 변경, 재추첨, 계열 변환에 사용하는 재료.
- **봉인 인장**: 원하는 옵션을 고정한 채 다른 옵션만 변경할 수 있게 해주는 보호 재료.
- **태그 에센스**: 특정 옵션 계열의 출현 확률을 높이는 테마 재료.
- 재료는 범용 재료와 테마형 재료로 나뉘며, 플레이어는 원하는 빌드에 맞는 사냥 목표를 스스로 정하게 된다.

## 플레이어 동기
- **단기 동기**: 원하는 옵션 한 줄을 만들거나, 현재 핵심 옵션을 한 단계 강화하고 싶다.
- **중기 동기**: 특정 태그 조합을 완성해 전투 성능이 눈에 띄게 달라지는 빌드를 만든다.
- **장기 동기**: 장비 수는 적지만 서로 다른 빌드 아키타입을 별도 세팅으로 보유하고 실험한다.
- 이 동기는 프로젝트의 핵심 구조인 **아웃게임 준비 -> 인게임 검증 -> 보상 -> 다음 준비**를 직접 강화한다.

## 현재 프로젝트와의 적합성
- 2분 내외 전투에서도 옵션 조합 차이가 즉시 드러나므로 짧은 세션 구조와 잘 맞는다.
- 전투 중 레벨업이나 즉석 선택 없이도 **사전 빌드 준비의 가치**를 강하게 보여줄 수 있다.
- 장비, 옵션, 재료를 분리한 구조는 `EquipData`, `Modifier`, `Tag` 중심의 데이터 주도 설계와도 잘 맞는다.

## Data Model 초안

| 모델 | 역할 | 핵심 필드 |
| --- | --- | --- |
| `EquipTemplate` | 장비 원형 데이터 | `equipId`, `slotType`, `name`, `baseStats`, `optionSlotCount`, `allowedOptionTags`, `implicitTrait` |
| `EquipInstance` | 유저가 실제 보유한 장비 인스턴스 | `instanceId`, `templateId`, `baseStatRolls`, `optionInstances`, `lockStates`, `upgradeLevel` |
| `OptionTemplate` | 옵션 정의 데이터 | `optionId`, `name`, `category`, `tags`, `valueRange`, `tierRange`, `exclusiveGroup`, `applicableSlots`, `weight` |
| `OptionInstance` | 장비에 부착된 개별 옵션 상태 | `optionInstanceId`, `optionId`, `tier`, `rolledValue`, `reinforceLevel`, `isLocked` |
| `MaterialItemData` | 재료 아이템 정의 | `materialId`, `name`, `materialType`, `relatedTags`, `grade`, `stackLimit` |
| `CraftActionData` | 옵션 생성, 강화, 변경 규칙 | `actionId`, `actionType`, `costMaterials`, `costCurrency`, `targetRule`, `resultRule` |
| `LoadoutData` | 전투 전 빌드 저장 | `loadoutId`, `characterId`, `equipInstanceIds`, `buildTags`, `lastTestResult` |
| `RewardTableData` | 전투 보상 분배 규칙 | `rewardTableId`, `stageId`, `dropMaterials`, `tagBias`, `scoreMultiplier` |

## 컨셉 결론
**리릭 포지 아레나**의 장비 시스템은 “많은 장비를 줍는 재미”보다, **적은 장비를 끝까지 깎고 조정해 완성형 빌드를 만드는 재미**에 집중한다.  
그 결과 전투는 짧아도 반복 욕구가 강하고, 파밍은 단순 수집이 아니라 **옵션 설계와 성능 검증의 루프**가 된다.
